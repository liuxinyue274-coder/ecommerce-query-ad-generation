# 演示模式重做 + "最终文案"概念纠正

**日期**: 2026-05-24
**作者**: 设计师 + Claude
**目标文件**: `.\app.py`

## 背景

之前一版演示模式(2026-05-24 早些时候)把 C 端预览作为现有调试 UI 的一个新增区块,导致两个问题:

1. **重复的 query 入口**:页面顶部一个"输入 Query + ▶ 运行"调试风输入,C 端预览里又有一个装饰性的"🔍 搜索"框,搜索框点击没反应。两套并存视觉冗余,也不沉浸。
2. **"最终广告文案"概念错误**:V5 输出的 `final_ad_copy` 是 ranker Top1,看板把它放在显眼的绿色 success 框里,误导成"系统的最终产物是一条文案"。但本系统的实际产物是「一组 (商品, 文案) 配对」—— 每个召回商品配它自己的广告文案,没有"唯一最终文案"这种东西。

## 设计

### 1. 演示模式 UI(全面重做)

页面布局:

```
┌─────────────────────────────────────────────────────┐
│ 🎯 Ad Studio              [🔧 调试] [🛒 演示]        │
└─────────────────────────────────────────────────────┘

(sidebar 隐藏)

┌─────────────────────────────────────────────────────┐
│ ←  🔍 [运动手环          ]  [provider▽]  [搜索]      │  ← 真 text_input
│   综合  销量  价格 ↑  筛选 ▽                          │
└─────────────────────────────────────────────────────┘

🔥 热门搜索: [运动手环] [宿舍吹风机] [通勤双肩包] ...

[sac.steps 链路进度: 意图 → 证据 → 候选 → 配对]

┌─ 商品列表(3 列网格) ──────────────────────────────┐
│ [⌚]                                                │
│ **跑步通勤都能戴,心率睡眠随手看。**     ← LLM 文案,主标题
│ 小米手环 8 NFC 版多功能运动手环…         ← 原 item_title,小字灰
│ 品牌 · 类目                                          │
│ ★★★★☆ 4.5 · 已售 8957+                              │
│ ¥156                                                 │
│ [立即购买] [加购物车]                                │
└─────────────────────────────────────────────────────┘
```

**关键变更**:

- 顶部 Taobao 风格搜索框是**真 query 输入**,不再是装饰 markdown
- 「搜索」按钮 / Enter 触发 V5(等价于调试模式的 ▶ 运行)
- provider 下拉嵌在搜索框右侧、搜索按钮左侧,方便演示中途切 mock
- 5 个示例按钮变成搜索框下方的「🔥 热门搜索」标签,点击=填入 query 并立即跑
- 「←」返回箭头作为装饰保留(不可点),维持淘宝氛围
- sidebar 用 CSS 隐藏(`section[data-testid="stSidebar"] {display:none}`),无需 rerun
- 删除独立的「最终广告文案」绿色 success 框
- 删除独立的「候选文案 Top3」精简区
- 删除商品卡内的 💡 文案小框(因为文案已经升级为主标题)
- Query 多样性增强 expander 在演示模式隐藏

### 2. 商品 ↔ 文案 映射规则

每个召回商品 `item_id` 对应一条文案,优先级:

1. **首选**:`ranked_candidates` 中引用了该 item 的候选,挑 `rank_score` 最高的那条 → 用其 `text`
2. **兜底**:如果没有候选引用该 item,用 `ranked_candidates[0].text` 作为通用文案,样式上**弱化**(灰色边框 + "通用文案"小标签)
3. **缺失**:如果连候选都没有,该商品卡不显示文案位

实现:重写 `render_consumer_view` 内部的 `copy_by_item` 计算逻辑(目前是"第一个引用就赢",改成"最高 rank 赢")。

### 3. 调试模式纠正"最终文案"概念

**删除**:`render_final_copy(result)` 那个绿色 success 框。

**新增** `render_pair_table(result)`,作为调试模式顶部的结果区:

```
### 🎯 系统产物:商品 ↔ 文案 配对(N 条)

[Streamlit dataframe]
┌────┬──────────────────────┬─────────────────────┬──────┬──────┐
│ #  │ 商品 (item_title)    │ LLM 文案            │ rank │ valid│
├────┼──────────────────────┼─────────────────────┼──────┼──────┤
│ 1  │ 小米手环 8 NFC 版…   │ 跑步通勤都能戴…     │ 4.32 │ ✅   │
│ 2  │ 象牙白多功能…        │ 心率睡眠随手看…     │ 3.87 │ ✅   │
│ 3  │ 情侣款蓝牙手表…      │ (通用兜底文案)      │ 3.40 │ ✅   │
└────┴──────────────────────┴─────────────────────┴──────┴──────┘

caption: V5 实际产出 N 条候选文案,通过 used_evidence 关联到具体商品。
        下方为系统中间产物。
```

调试模式新结构(自上而下):

1. `render_pipeline_steps` — sac.steps 链路进度(已有,保留)
2. `render_pair_table` — **新顶部结果区**(替代 `render_final_copy`)
3. `render_status_strip` — 状态条(provider/model/fallback/cache,已有)
4. `🛒 C 端预览` expander(默认折叠,内部调用 `render_consumer_view`)
5. `🛍️ 全部候选文案诊断`(原 `render_copy_product_pairs`,**重命名**)
6. 意图 + 证据双列(已有)
7. 候选广告文案 Top3(已有,带 subscores 等诊断细节)
8. Validator / 高级面板 / 完整 JSON / Badcase 表单(已有)

### 4. 状态共享

- `query` 在演示模式和调试模式之间共享 `st.session_state["query"]`
- `provider` 同样共享 `st.session_state["provider"]`(sidebar 与演示模式头部下拉绑同一 key)
- 切换模式时 query / provider / 上次结果都保留,体验连续

2. `streamlit run app.py`,手动 smoke:
   - 调试模式输入"运动手环"运行,确认顶部出现配对表、不再有"最终广告文案"绿框
   - 切到演示模式,确认 sidebar 消失
   - 在演示模式搜索框输入"宿舍吹风机"按 Enter,跑通 V5,商品卡片 LLM 文案在主标题位置、原 item_title 在副标题位置
   - 点 🔥 热门搜索某个标签,query 自动填入并跑通
   - 切回调试模式,确认 sidebar 回来,query 还是上次的"宿舍吹风机"
